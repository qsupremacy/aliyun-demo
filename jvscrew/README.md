# same.log vs diff.log 性能对比分析

## 1. 测试场景概述

| 维度 | same.log | diff.log |
|------|----------|----------|
| 测试模式 | 共享 user/session | 独立 user/session |
| 请求次数 | 10 | 10 |
| ExternalUserId | `haolipeng_jvs_crew` (同一) | 10 个不同 `user-xxxxxx` |
| SessionId | `11fbd74b-…` (同一) | 10 个不同 Session |
| 提示语 | "hello，不要思考，直接返回我hello" | 同左 |
| 服务端 Trace | ✓ (TurnCount=10, 单一 Session) | ✓ (每个请求独立 Turn) |
| 成功状态 | 100% (status=200) | 100% (status=200) |
| 模型 | pro | pro |

> **核心区别**: same 模式复用 user/session,服务端 LLM 上下文可缓存;diff 模式每次都是全新 user/session,服务端需全量构建会话状态。

## 6. 关键发现

### 6.1 same.log 优势
- ✅ **客户端耗时低 47.3%** (4.14s vs 7.85s)
- ✅ **服务端总耗时低 44.6%** (2.76s vs 4.98s)
- ✅ SkillSync 阶段短 87% (368ms vs 2783ms) — 复用会话状态

### 6.2 diff.log 代价
- ❌ SkillSync 阶段剧增 (2530~3004ms) — 每次都需加载/构建会话上下文
- ❌ 客户端开销更大 (~2.87s 网络+处理,vs same 的 ~1.38s)
- ❌ 用户隔离更彻底,但延迟更高

### 6.3 异常请求

**same.log 异常**:
- Turn 2: client 6.106s, server 4.860s — LLM 突增至 4326ms (服务端波动)
- Turn 5: server 3.241s — LLM 2891ms

**diff.log 异常**:
- Turn 8: client 8.746s,但 server 仅 4.230s — 客户端网络/排队问题
- Turn 9: client 10.101s — 同时 LLM (2445ms) + LAUNCH (3117ms) 偏高

## 7. 性能瓶颈定位

### same.log 瓶颈: **LLM 调用 (80.5%)**
- LLM DurationMs 是最大耗时项
- SkillSync 短 (368ms),会话复用生效
- 优化方向: LLM 流式响应 / 更轻量模型 / 结果缓存

### diff.log 瓶颈: **SkillSync (55.9%)**
- 每次新建 Session 触发完整 SkillSync (~2.8s)
- 加载工作区、检查点、技能包等
- 优化方向:
  - 复用 SessionId (应用层缓存)
  - 缩短 SkillSync 流程 (懒加载、按需同步)
  - 预热常用 Session

---

## 8. 结论与建议

### 结论
| 维度 | 共享模式 (same) | 独立模式 (diff) |
|------|----------------|----------------|
| 端到端性能 | **优** (4.14s) | 差 (7.85s) |
| 服务端开销 | **优** (2.76s) | 差 (4.98s) |
| 稳定性 (LLM) | 中 (波动大) | **优** (波动小) |
| Credit 消耗 | 接近 (5.35) | 接近 (5.32) |
| 用户隔离性 | 弱 | **强** |

### 建议
1. **高频调用场景** → 采用 same 模式 (复用 user/session),可获 ~48% 性能提升
2. **SkillSync 优化** → 重点投入,diff 模式下占总耗时 56%
3. **LLM 调用优化** → same 模式下需重点优化,可考虑流式响应 + 缓存
4. **客户端网络优化** → 两模式均有 1.4~2.9s 的网络+客户端开销,可通过 HTTP/2、长连接、批量请求降低
5. **监控告警**:
   - same: LLM > 4s 告警
   - diff: SkillSync > 3s 告警
   - 客户端 vs 服务端差异 > 2s 时排查网络
