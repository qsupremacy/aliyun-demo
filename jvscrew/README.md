# jvscrew测试日志分析

## 摘要: 结论与建议

### 术语说明

| 术语 | 定义 | 本场景中包含的耗时 |
|------|------|-------------------|
| **端到端性能** | 客户端发起请求到收到完整响应的总耗时 (即 `[耗时]` 行) | 客户端处理 + 网络传输 + 服务端全流程 + 反序列化 |
| **服务端开销** | 服务端处理一次请求的总耗时 (服务端 Trace 中 `LAUNCH.TotalMs + LLM.DurationMs`) | LAUNCH 阶段 + Guardrail + LLM 推理 |
| **空载耗时** | **客户端耗时 − LLM 模型耗时**,代表"非 LLM 计算"的所有固定开销 | 客户端签名/构造 + 网络传输 + 服务端 LAUNCH + Guardrail 等 |
| LLM 推理 | 模型生成 token 的耗时 (`smartclaw.llm.call` DurationMs) | 仅模型推理本身 |
| Credit | 服务端计费单位,与 LLM 调用量正相关 | — |

> **空载耗时的意义**: 剥离 LLM 推理的"业务不可控"波动后,聚焦反映系统本身的固定开销与可优化空间,常用于评估缓存复用、网络优化、Session 管理等非模型层面的改进效果。

### TL;DR
- **共享 Session 模式 (same) 性能显著优于独立 Session 模式 (diff)**: 端到端耗时 **4.14s vs 7.85s** (-48%), 空载耗时 **1.90s vs 5.85s** (-68%)
- **真正的瓶颈不是 LLM, 而是空载开销** — 即使消除 LLM, same 仍比 diff 快 3.95s
- **diff 模式的最大空载来自 SkillSync (~2.78s)**, 即每次新建 Session 的固定成本
- **建议**: 高频场景复用 SessionId (same),并重点优化 SkillSync 与客户端网络开销

### 结论对比表

| 维度 | 共享模式 (same) | 独立模式 (diff) |
|------|----------------|----------------|
| 端到端性能 | **优** (4.14s) | 差 (7.85s) |
| 服务端开销 | **优** (2.76s) | 差 (4.98s) |
| LLM 推理 | 慢 (2.24s, 波动大) | **优** (2.00s, 稳定) |
| **空载耗时** | **优 (1.90s, 46.0%)** | 差 (5.85s, 74.6%) |
| 稳定性 (LLM) | 中 (波动大) | **优** (波动小) |
| Credit 消耗 | 接近 (5.35) | 接近 (5.32) |
| 用户隔离性 | 弱 | **强** |

> **关键洞察**: 即使消除 LLM 耗时 (空载视角),same 模式仍比 diff 模式快 **3.95s**,占绝对优势。LLM 不是主要瓶颈,空载开销才是 — 特别是 SkillSync 在 diff 模式下贡献了 ~2.78s 的固定开销。

### 行动建议 (按优先级)

1. **复用 SessionId (最高优先级)**: 高频调用场景采用 same 模式,端到端性能提升 ~48%,空载开销降低 67.5%
2. **SkillSync 优化**: diff 模式下占空载 47.6%,可通过复用 SessionId、按需加载、预热常用 Session 降低
3. **LLM 调用优化**: same 模式下波动大 (Turn2 达 4326ms),可考虑流式响应 + 结果缓存
4. **客户端网络优化**: same 模式仍有 ~1.38s 的客户端+网络空载 (占空载 72.5%),可通过 HTTP/2、长连接、复用 Token 降低
5. **监控告警 (重点关注空载耗时)**:
   - 空载耗时 > 6s (diff) / > 3s (same) → 立即排查
   - same: LLM > 4s 告警
   - diff: SkillSync > 3s 告警
   - 客户端 vs 服务端差异 > 2s 时排查网络
   - 空载耗时长期高于 P95 基线 1.5x → 资源争抢或 GC 抖动

## 6. 关键发现

### 6.1 same.log 优势
- ✅ **客户端耗时低 47.3%** (4.14s vs 7.85s)
- ✅ **服务端总耗时低 44.6%** (2.76s vs 4.98s)
- ✅ SkillSync 阶段短 87% (368ms vs 2783ms) — 复用会话状态

### 6.2 diff.log 代价
- ❌ SkillSync 阶段剧增 (2530~3004ms) — 每次都需加载/构建会话上下文
- ❌ 客户端开销更大 (~2.87s 网络+处理,vs same 的 ~1.38s)
- ❌ 用户隔离更彻底,但延迟更高

### 6.3 异常请求

**same.log 异常**:
- Turn 2: client 6.106s, server 4.860s — LLM 突增至 4326ms (服务端波动)
- Turn 5: server 3.241s — LLM 2891ms

**diff.log 异常**:
- Turn 8: client 8.746s,但 server 仅 4.230s — 客户端网络/排队问题
- Turn 9: client 10.101s — 同时 LLM (2445ms) + LAUNCH (3117ms) 偏高

## 7. 性能瓶颈定位

### same.log 瓶颈: **LLM 调用 (80.5%)**
- LLM DurationMs 是最大耗时项
- SkillSync 短 (368ms),会话复用生效
- 优化方向: LLM 流式响应 / 更轻量模型 / 结果缓存

### diff.log 瓶颈: **SkillSync (55.9%)**
- 每次新建 Session 触发完整 SkillSync (~2.8s)
- 加载工作区、检查点、技能包等
- 优化方向:
  - 复用 SessionId (应用层缓存)
  - 缩短 SkillSync 流程 (懒加载、按需同步)
  - 预热常用 Session
