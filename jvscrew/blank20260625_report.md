## 关键发现

1. **LLM 调用是性能瓶颈**
   - 占服务端总耗时 **80.5%**
   - 范围 1346~4326ms，标准差大 (P50 ≈ 1.6s, P95 ≈ 4.3s)

2. **SkillSync 是 LAUNCH 阶段主要开销**
   - 占 LAUNCH 阶段 **70.2%** (368.5/525.1)
   - 范围 300~427ms，相对稳定

3. **Guardrail 检查极快**
   - input/output 敏感词检测均 ≤ 0ms (本次未触发)
   - 不会成为性能瓶颈

4. **Turn 2 异常慢**
   - LLM 调用 4326ms (平均的 1.93x)
   - 推测后端 LLM 服务偶发抖动

5. **客户端开销 ~1.4s/请求**
   - 占客户端总耗时 33%
   - 主要为网络 + 客户端签名/构造
   - 优化空间: 复用 HTTP 连接 / 减少中间环节

## 优化建议

1. **LLM 调用优化** (收益最大)
   - 启用 LLM 流式响应 (TTFT 优化)
   - 短 prompt 场景可使用更轻量模型 (当前为 pro)
   - 增加 LLM 调用缓存 (相同 prompt 复用结果)

2. **LAUNCH 阶段**
   - SkillSync 占 LAUNCH 70%，可考虑延迟同步或预热
   - 减少 WorkspacePreflight 检查频次

3. **客户端优化**
   - 减少 ~1.4s 的网络/处理开销
   - 启用 HTTP/2 或长连接复用

4. **监控告警**
   - LLM DurationMs > 4s 时告警
   - 客户端 vs 服务端耗时差异 > 2s 时排查网络
