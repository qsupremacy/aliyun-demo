### 原因分析

#### 1. 并发竞争 (最可能)
- 多个并发请求同时竞争有限的服务端资源
- 10 个请求同时出现 2.4-2.5s 延迟，呈簇状分布，强烈暗示资源争抢

#### 2. 连接池耗尽
- 并发数超过 HTTP 连接池大小
- 等待连接的请求被阻塞，直到连接释放

#### 3. 服务端限流
- 高并发触发阿里云 API 速率限制
- 请求进入排队或被延迟处理

#### 4. 冷启动效应
- 函数计算冷启动导致首次并发请求延迟

## 结论与建议

### 结论

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| 高吞吐量需求 | multi.log (并发) | QPS 高 40 倍 |
| 延迟稳定性优先 | run.log (顺序) | 延迟波动小 |
| 生产环境建议 | 并发 + 优化 | 兼顾吞吐和稳定 |

### 慢请求优化建议

1. **连接池调优**
   - 增加 HTTP 连接池大小
   - 配置合理的 maxConnections

2. **请求分批**
   - 将大批量并发请求拆分为小批次
   - 每批 50-100 请求，间隔 100-200ms

3. **超时重试**
   - 设置合理的超时时间 (建议 5s)
   - 对超时请求自动重试

4. **限流保护**
   - 客户端设置 QPS 上限
   - 使用令牌桶/漏桶算法控流

5. **监控告警**
   - 对 >1s 请求设置实时告警
   - 记录慢请求详情用于分析
